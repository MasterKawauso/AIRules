# WORKFLOW — 分担・長期作業フロー

設計を伴う実装、タスク分割・並列実装時に適用。軽微な単発修正は対象外。各実装は`AGENTS.md`に従う。

## 基本フロー

1. 担当AI・モデル・思考深度が未指定で、選択が品質・費用・時間を大きく変える場合だけ、推奨案付きで確認する。それ以外は既定担当を使う
2. 要件→設計→分割→実装→成果物レビュー→統合→統合レビュー→完了の順に進む。整理済みの`PLAN.md`は要件、設計文書は設計工程を兼ね、不足部分だけ補う
3. 独立性と統合点が明確な作業だけ1〜5 Workerへ分割する。各成果物は1回レビューし、問題箇所だけ修正・再確認する。Codexは成果物別レビューを統合レビューへまとめてよい
4. 統合レビューは統合差分と接続点だけを対象にする。Public API/データ構造、破壊的変更、セキュリティ、広範囲・高リスク変更はCodex↔Claudeで相互レビューし、それ以外は単独レビューでよい。往復は2回まで
5. 相互レビューは変更差分、仕様、未決事項だけを渡す。Claude→Codexは`codex:rescue`、Codex→Claudeは`Get-Command claude`確認後`claude -p "<レビュー依頼>"`。必須レビューが利用不能なら停止してユーザーへ依頼する

## 役割

- Architect: 仕様・設計、分割、完了条件
- Worker: 担当範囲だけ実装。範囲外の仕様・設計変更や大規模リファクタは禁止し、不明点を報告
- Reviewer: 担当差分を`REVIEW.md`で確認し、実害のある問題だけ示す
- Integrator: 統合、接続漏れ・競合・仕様差異、必要最小限の統合修正、最終レビュー

## 長い会話の引継ぎ

残り文脈では安全な完了が難しい、会話の要約・圧縮が発生した、または長い調査出力で重要情報が埋もれ始めた場合に限り、作業の安全な区切りでプロジェクト直下の`SESSION.md`を新規作成または更新する。進行中の不可分な変更・復旧操作は区切りまで終える。

`SESSION.md`には次だけを簡潔に記す。

- 目的・完了条件
- 完了した成果と重要な決定
- 現在の状態、変更ファイル、検証結果
- 残作業を再開順に列挙し、次に実行する確認・コマンドを明記
- 未決事項、阻害要因、注意点

会話全文、既存`PLAN.md`/`PROGRESS.md`の転載、秘密情報、不要な試行履歴は含めない。保存後、ユーザーへ「新しいセッションで`SESSION.md`を読んで続行」と切替を促す。新セッションは同ファイルを1回読み、実際の差分・状態を確認してから再開する。完了済み内容を再実行しない。恒久的な成果は`PROGRESS.md`、一時的な引継ぎだけを`SESSION.md`に置く。
