# AGENTS.md — 共通開発ルール（正本）

Codex/Claude Code共通の常時ルール。条件付きルールは`airules/`にあり、必要な文書だけをセッション中1回読む（内容が既に文脈にあれば再読不要）。

## 役割・優先順位

- 既定担当: Codex=実装・修正・差分整理、Claude=設計・仕様・レビュー・ゲームエンジン補助。ユーザー指示を優先し、引継ぎは`PLAN.md`/`PROGRESS.md`/`docs/DESIGN_NOTES.md`を使う
- 優先順位: ユーザー指示 > プロジェクト`AGENTS.md` > `PLAN.md` > 本書 > `airules/`。ツール既定動作は順位外で、本書より優先しない

## セッション開始時（初回のみ）

存在する`PLAN.md`・`PROGRESS.md`を読み、フェーズと該当ルールを確認する。認識報告は不一致・作業影響がある場合だけ一行で行う。

## 条件付きルール（作業前に必読）

| 条件 | ファイル |
|---|---|
| 新規機能・仕様変更・要件定義 | REQUIREMENTS.md |
| 企画検証・市場性・価格・販売戦略・プロダクト判断 | 導入済みのPM Skillsから該当スキル |
| 実装/修正/デバッグ | PITFALLS.md、THINKING.md |
| 設計/実装計画 | THINKING.md、DESIGN.md |
| 設計を伴う実装・分担・並列作業 | WORKFLOW.md |
| Unity/UE/Godot判定 | UNITY.md/UE5.md/GODOT.md |
| ゲーム実装・修正全般 | GAME_COMMON.md |
| レビュー・完了報告前 | REVIEW.md |
| Git操作 | GIT.md |
| PR作成・レビュー/GitHub Issue管理 | GITHUB.md |

## 基本原則

推測で実装・回答せず、案内前に実コードを確認する。調査不能な不明点だけ候補を添えて確認し、単純な解決を優先する。企画や主要仕様では、実装可能性だけでなく対象顧客・価値仮説・競合・価格・市場投入・検証方法が論点なら、一般論で済ませず導入済みのPM Skillsを使う。

## 開発フェーズ

`PROGRESS.md`冒頭で確認し、なければモック期とする。モック期は速度・遊べる検証、本実装期は保守・拡張性を優先（`GAME_COMMON.md`準拠）。捨てにくい共通機能は雑に直書きせず、移行はユーザー指示時のみ行う。

## 実装・報告

- 命名: メンバー変数はアンダースコアなし小文字始まり
- 完了条件と変更予定を短く定め、設計意図を保つ最小差分を1機能ずつコンパイル可能に実装し、その差分を1回セルフレビューする
- 進捗報告は開始時、長時間作業の節目、前提変更・問題発生時に短く行う。同じ計画や確認済み事項を繰り返さない
- 指示外のファイル・改名・コメント/import整理・整形だけの変更・大規模リファクタは禁止
- 設計/Public API/データ構造変更、外部ライブラリ追加、`PLAN.md`外の機能追加は事前確認する
- 仕様上の区切りで`PROGRESS.md`へ簡潔に追記し、フェーズ表記を維持する。実行可能な後回し・新発見課題だけを重複なく`GITHUB.md`に従いIssue化する
- 完了報告は実装、検証・レビュー、参照ルール、懸念・残タスク、`PROGRESS.md`追記を簡潔にまとめ、該当なしは省く
